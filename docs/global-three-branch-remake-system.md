# Global Three-Branch Remake System

本文件沉淀复刻 / 原型重构系统的三条交付支线通用规则。它不绑定某一种题材，可用于角色动作、产品广告、文旅空间、建筑、汽车、食物、UI、卡通、怪物、超现实短片等不同类型。

## 1. 通用前置规则

在选择 A/B/C 线之前，必须先完成后台 DNA 拆解和审核：

- 剧情功能和情绪曲线
- 镜头数量、镜头顺序、切点节奏
- 运镜、角度、构图、镜头质感
- 主体出场、身份确认、表演节拍
- 台词 / 旁白功能和时机
- 道具 / 工具 / 机制触发
- 群体或环境反应
- 变身、展开、损坏、激活、完成等状态变化
- VFX 触发、扩散、回拉、升级和终态
- 音效、BGM、字幕和后期边界
- 最终 payoff 和尾部可读时间

最终模型提示词只描述新片本身，不写“参考片、原片、学习、复制、后台分析”等过程语言。

## 2. A 线：资产直出

A 线用于用户提供资产、要求资产一致，或身份 / 产品 / 场景 / 道具稳定性优先的复刻重构。

核心模型：

```text
身份资产 -> 环境资产 -> 道具/机制资产 -> 执行态资产 -> 最终 payoff 资产
```

资产设计规则：

- 每张上传图只承担一个窄职责。
- 身份资产保持静态干净，只锁脸、轮廓、产品形状、服装 DNA、品牌物、载具形态、材质系统或生物结构。
- 执行态资产只表达一个允许状态，例如变身中、产品展开、手部交互、撞击态、破损态、UI 激活态、食物切开态、车辆漂移动作等。
- 最终 payoff 资产必须单独设计，只允许在最终镜头出现，例如最终法相、产品 hero reveal、品牌揭示、完成变身、爆炸花、群众揭示、地标揭示、成品菜、完成 UI 状态。
- 多状态板可以用于人审，但不要直接上传给 Seedance，除非你真的想让多个状态同时出现在画面中。
- 不要默认用关键帧解决时点问题；关键帧可能泄漏到错误镜头。优先使用资产权限和时间权限语言。
- 单条 12-15 秒 A 线无法同时保住快节奏和资产一致时，应拆成多段生成再剪辑。

提示词写法：

- 先写上传顺序和资产权限。
- 使用 `@图片X 只锁身份`、`@图片Y 只允许用于 SH06`、`@图片Z 不得在 SH08 前出现`、`不要把该资产渲染成第二主体` 这类权限语句。
- 写镜头职责，而不只是画面描述：引入、确认、触发、反应、变化、冲击、揭示、收束。
- 状态变化必须写成过程：`变化前状态 -> 外力或内力作用 -> 材质/身体/物体/机制变化 -> 新状态稳定`。
- 负面词按当前类型定制，不把玄幻、产品、建筑、食物、UI、卡通的失败项混贴。

A 线基准：

- 当前系统把 86/100 作为有资产复刻的最低可用视觉基准。
- 交付提示词应按 95/100 标准自审。
- 如果资产稳定但时间整体后移，不默认继续加图；先减少资产、隔离职责、强化时间权限，仍失败时拆段。

## 3. B 线：故事板控制

B 线用于镜头顺序、切点节奏、动作方向、构图重心和空间调度优先的复刻重构。

故事板规则：

- Seedance 只接收干净黑白手稿故事板或干净控制帧。
- 不上传带箭头、标签、编号、文字、彩色标记、标题栏、UI 边框或白底资产板痕迹的故事板。
- 参考片有 8 个镜头，就设计 8 格或 8 张干净控制帧，不强行变成 12 格。
- 超过 10 镜必须拆成多个故事板或多个 Seedance 段落。
- 带标注故事板只用于人审，Seedance 使用无标注版本。
- Seedance 提示词必须说明故事板只锁镜头顺序、构图、运动方向和切点节奏，不渲染边框、编号、箭头、文字或面板布局。

可复用表述：

```text
将提供的 [clean board name] / [grid size] / SH01-SH08 制作成一段流畅的电影级动画视频。严格保持 SH01-SH08 的镜头顺序、切点节奏、构图重心、动作方向和镜头连贯性。不要渲染故事板边框、编号、箭头、文字标注或面板布局。故事板只作为运动、构图、镜头切点和动作方向参考，最终必须是一条连续成片。
```

B 线常见失败：

- 箭头、标签、编号或面板 UI 被渲染进画面。
- 成片变成故事板翻页或网格。
- 角色身份跟随草图而不是资产图。
- 切点节奏变成慢推慢摇。
- 提示词和控制图过多，互相污染。

最佳实践：

- 需要身份稳定时，用 A 线资产 + B 线干净故事板。
- 身份参考和运动参考必须分开写。
- 如果故事板压过角色资产，将故事板简化成黑白剪影，并强化资产身份权限。

## 4. C 线：纯文本直出

C 线用于用户只给片段、短片或链接，没有资产和故事板，或明确只要一条 Seedance 提示词快速测试方向。

C 线规则：

- 无 `@` 引用、无上传图片、无资产、无故事板。
- 提示词通常控制在用户要求的长度内，常见为 2500 中文字以内。
- 短不等于虚，每句话都要控制剧情功能、镜头时机、运镜、表演、状态变化、VFX、声音或失败预防。
- 负面词只写纯文本生成风险：镜头数不足、一镜到底、近脸延后、口型缺失、状态变化缺失、触发点缺失、最终态提前、身份漂移、道具漂移、坏手、风格漂、字幕文字等。
- 不写 A/B 线的资产板、故事板、箭头、面板、`@` 引用逻辑，除非用户真的提供了这些输入。
- 用因果锁，不只用时间锁。过度激进的时间闸门会导致顺序混乱或最终态提前。

C 线因果模型：

```text
入场/建立 -> 身份/表演/台词节拍 -> 道具/工具/机制触发 -> 环境/群体反应 -> 状态变化 -> 触发/冲击 -> 最终 payoff
```

C 线交付前自检：

- 纯文本，无 `@` 引用。
- 不出现源片、学习、复制等语言。
- 符合字数限制。
- 镜头职责齐全。
- 状态变化明确。
- 最终 payoff 存在且有尾部时间。
- 不生成后期字幕和 BGM。

## 5. 分支选择

- 身份、产品、道具或场景一致性最重要：选 A 线。
- 镜头顺序、切点节奏、构图方向最重要：选 B 线。
- 没有资产，只想快速拿一条 Seedance 提示词：选 C 线。
- 身份和镜头节奏都重要：选 A+B。
- 单条 12-15 秒无法兼顾快节奏和复杂状态：拆段生成再剪。

## 6. 跨支线经验

- 不要用堆字解决所有问题，先判断失败类型：资产漂移、时间漂移、状态漏项、最终 payoff 太短、源片语言泄漏、故事板污染或节奏崩塌。
- 不要用更多关键帧解决所有时间问题，关键帧也会泄漏。
- 不要把多状态图作为视频输入，除非想同时展示多状态。
- 始终分离身份、动作状态和最终 payoff。
- 台词和字幕要分开，台词可以是口型/声音表演，字幕默认不生成。
- 后期 BGM 不进入 Seedance；只写 `无BGM` 和必要 SFX。
- 最终提示词不是分析记录，而是一个新独立短片的生产命令。

## 7. 最低交付标准

任何复刻 / 重构系统输出都必须：

- 标明 A/B/C 或混合支线。
- 写清上传 / 输入逻辑。
- 保留参考片的功能镜头数量和因果结构。
- 后台明确变量和不变量。
- 交付 source-free 的模型提示词。
- 交付前执行 prompt lint。
- 测试结果要用证据图、切点和失败类别评分，不能用泛泛感受修提示词。
